# 火星车教学计划

这份计划适合家长陪孩子分阶段完成项目。目标不是尽快做出成品，而是让孩子真正理解每一步为什么这样做。

## 总体原则

1. 一次只学一个新概念。
2. 一次只增加一个新硬件模块。
3. 每次实验都先验证最小结果，再进入下一步。
4. 允许孩子先试错，再一起复盘。

## 阶段 1：认识开发环境

目标：

- 知道 Pico 是什么。
- 知道 VS Code 是写代码的地方。
- 知道程序可以上传到 Pico 上运行。

建议时长：

- 20 到 30 分钟

操作内容：

1. 安装 VS Code。
2. 安装 MicroPico 插件。
3. 认识命令面板。
4. 解释什么是“上传到板子”。

建议讲法：

- 电脑像写作业的桌子。
- Pico 像一台很小的电脑。
- 代码写在电脑里，再送到 Pico 里执行。

完成标准：

- 孩子知道如何打开项目。
- 孩子知道如何打开命令面板。

## 阶段 2：刷固件并连接 Pico

目标：

- 学会让 Pico 进入可运行 MicroPython 的状态。
- 学会在 VS Code 中连接设备。

建议时长：

- 20 分钟

操作内容：

1. 下载 MicroPython 固件。
2. 按住 BOOTSEL 接入 USB。
3. 拖入 uf2 固件文件。
4. 在 VS Code 中执行 `MicroPico: Connect`。
5. 观察 REPL 中出现 `>>>`。

关键知识点：

- 固件像“系统软件”。
- REPL 是能和 Pico 直接对话的窗口。

完成标准：

- 能看到 `>>>` 提示符。
- 能在 REPL 里执行 `print("hello")`。

## 阶段 3：第一个成功程序

目标：

- 成功运行 blink.py。
- 建立“代码能控制硬件”的直觉。

建议时长：

- 15 到 20 分钟

操作内容：

1. 打开 blink.py。
2. 执行 `MicroPico: Run current file on Pico`。
3. 观察 LED 闪烁。
4. 让孩子自己把闪烁次数改掉，再运行一次。

建议提问：

1. 为什么 LED 会一亮一灭？
2. 哪一行代码控制等待时间？
3. 如果把 0.5 改成 1，会发生什么？

完成标准：

- 孩子能自己运行 blink.py。
- 孩子能自己改一个数字并看到结果变化。

## 阶段 4：认识函数和模块

目标：

- 理解“函数”是把动作包装成名字。
- 理解“模块”是把相关代码放进一个文件。

建议时长：

- 20 到 30 分钟

操作内容：

1. 打开 motor.py。
2. 逐个讲解 `forward()`、`backward()`、`turn_left()`、`turn_right()`、`stop()`。
3. 解释为什么 main.py 可以 `import motor`。

建议讲法：

- 函数像按钮，按不同按钮就执行不同动作。
- 模块像工具箱，把同类工具放在一起。

完成标准：

- 孩子能说出 motor.py 是干什么的。
- 孩子知道 main.py 为什么能调用 motor.forward()。

## 阶段 5：电机测试

目标：

- 成功驱动小车前进、后退、左转、右转。
- 理解“修改代码会改变现实中的动作”。

建议时长：

- 30 到 40 分钟

操作内容：

1. 先执行 `MicroPico: Upload project to Pico`。
2. 打开 main.py。
3. 运行 main.py。
4. 观察动作顺序。
5. 如果方向不对，修改 motor.py 后重新上传。

建议提问：

1. 如果前进变成了后退，应该改哪里？
2. 为什么改完代码后要重新上传项目？
3. 为什么 main.py 默认只跑一轮？

完成标准：

- 小车能完成一轮基础动作。
- 孩子知道 motor.py 和 main.py 分别负责什么。

## 阶段 6：学会停止和调试

目标：

- 学会停止 Pico 上的程序。
- 学会用最基础的方法排查问题。

建议时长：

- 15 到 20 分钟

操作内容：

1. 在 `MicroPico vREPL` 中按 `Ctrl+C` 停止程序。
2. 介绍 `Ctrl+D` 软重启。
3. 演示在代码里加入 `print()`。
4. 演示在 REPL 中单独运行一小段代码。

关键知识点：

- 调试不一定靠复杂工具。
- 先缩小问题，再去改代码。

完成标准：

- 孩子能自己停掉程序。
- 孩子知道出现问题时先看终端输出。

## 阶段 7：开始阅读样例骨架

目标：

- 理解项目还有哪些模块没完成。
- 明白后续任务不是“抄答案”，而是自己补全功能。

建议时长：

- 20 分钟

操作内容：

1. 打开 radar.py。
2. 解释 trig 和 echo 是什么。
3. 打开 receiver.py。
4. 解释高低电平输入和遥控接管的思路。

建议讲法：

- 这些文件现在像“题目 + 提示”。
- 真正的代码要靠自己一步步补出来。

完成标准：

- 孩子知道 radar.py 和 receiver.py 还没有做完。
- 孩子知道后面要补的是哪一部分。

## 阶段 8：超声波测距练习

目标：

- 学会读取时间差。
- 学会给等待过程加超时保护。

建议时长：

- 30 到 45 分钟

操作内容：

1. 接好 HC-SR04。
2. 在 radar.py 里补完 `get_distance()`。
3. 每完成一步就打印一次测量结果。

建议提问：

1. 为什么不能一直卡在 while 里等回波？
2. 为什么需要超时？
3. 测出来的数字单位是什么？

完成标准：

- 能稳定打印距离值。
- 距离变化和障碍物远近大致一致。

## 阶段 9：遥控接管练习

目标：

- 学会读取多个输入引脚。
- 学会把输入状态转换成控制命令。

建议时长：

- 30 到 45 分钟

操作内容：

1. 接好遥控接收板信号线。
2. 先打印 `read_states()` 的结果。
3. 再补完 `get_command()` 逻辑。
4. 验证按不同按键时，返回的命令是否正确。

建议提问：

1. 如果两个方向同时按下怎么办？
2. 为什么没有输入时要返回 None？

完成标准：

- 能正确识别前进、后退、左转、右转。
- 能处理冲突输入。

## 阶段 10：整合成完整火星车

目标：

- 把电机、超声波、遥控接管整合起来。
- 做出“自动避障 + 手动优先”的完整主程序。

建议时长：

- 45 到 60 分钟

操作内容：

1. 在 main.py 中加入自动巡航逻辑。
2. 距离小于阈值时自动避障。
3. 遥控器有输入时优先手动接管。
4. 所有循环里保留短暂 sleep。

完成标准：

- 小车能自动前进。
- 有障碍时会避让。
- 遥控器有输入时会优先响应人工控制。

## 每阶段结束后都可以问的复盘问题

1. 今天新学会了什么？
2. 哪一步最难？
3. 哪个错误最有意思？
4. 如果下次再做，你会先检查什么？

## 家长陪学建议

1. 不要急着替孩子改代码，先让他自己描述问题。
2. 不要一次讲太多概念，讲一个就让他马上试一次。
3. 出错是学习的一部分，不要把“跑通”当成唯一目标。
4. 多让孩子预测结果，例如“你觉得这次会前进还是后退”。
5. 每完成一个阶段，就让他自己用一句话解释刚才做了什么。
